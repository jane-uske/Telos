# Telos 收费方案（暂缓实现，先归档）

> 状态：**已定方向，暂不开发**。当前只做"导出移到服务端"这一前置（收费闸门的地基）。

## 已定

- **分产品权益**：Telos 的付费与 Kairos 独立，各算各的（受众不重叠）。
- **不做订阅**，按导出付费（买断式）。
- **同一份简历，导出价格递减**：
  - 第 1 次导出：1.99
  - 第 2 次导出：1.29
  - 第 3 次及以后：0.99（下限）
  - 简历修改后再导出，仍按 0.99（不回涨到 1.99）
- 付费闸门放在**服务端导出**这一步（前端收费=可绕过）。

## 未定 / 待办

- **币种 + 主要区域**（¥ 还是 $，国内为主还是海外）——决定支付渠道（微信/支付宝 vs Stripe）。
- 支付模块本身（下单、回调对账、退款）——**本期不做**。

## 待评估的改良（我提的建议，未拍板）

1. **预充值余额（wallet）**：把"支付"和"导出"解耦——用户先充一次，之后每次导出从余额**静默扣款**，不再每次走支付流程。理由：每次导出都弹一次微信/支付宝，操作成本远高于 0.99；且微额多笔手续费吃亏。递减价照走，只是从余额扣。
2. **改错别字缓冲**：同一份简历付费导出后 **24h 内重导免费**，过期再按 0.99。避免"刚导完发现打错字又收钱"的差评。

## 实现时的数据（Telos 后端，按 gateway user_id / resume_id 记）

- `export_count`（每份简历的已导出次数）→ 决定当次价格。
- `wallet`（若采纳预充值）→ 余额。
- `orders` → 订单/充值流水，供回调对账。
- 导出流程里加权益判断点：查价 → 扣费/扣额度 → 生成 PDF。
