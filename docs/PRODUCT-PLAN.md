## 1. 当前产品已经实现的功能

产品定位：帮工程师把零散经历整理成**可追溯的职业证据**，再针对具体岗位生成可信简历与求职材料，覆盖「经历 → 岗位 → 简历/问题 → 模拟 → 真实面试 → 复盘反哺」的闭环。技术栈：Next.js 16 App Router + React 19 + TypeScript，单一 Zustand store（`web/src/lib/store.ts`）驱动状态，路由用 `screen`/`tab` 两级 store 状态模拟（非真实 URL 路由），IndexedDB 本地持久化。

### 1.1 已实现的页面能力（11 个 tab + 首页/登录）

| 页面（tab） | 核心能力 |
|---|---|
| `dashboard`（开始准备） | 问候语、首次引导 chips（8 步，仅首次展示）、「下一步」行动引擎（`useNextAction`，按优先级判断当前岗位最该做的一件事）、4 项统计卡、当前岗位列表、最近一次复盘摘要、待完善经历、待准备问题 |
| `import`（导入旧简历） | 粘贴/填示例简历文本 → AI（或确定性兜底）拆解成结构化经历段 → 逐条「转为证据卡」加入证据库 |
| `interview`（AI 访谈补全经历） | 选择一张未确认证据卡或发起「自由访谈」→ 多轮连续追问（背景/职责边界/关键决策/难点/协作/量化结果/证据）→ 支持 Agent 模式：`search_evidence`（核实说法有无证据支撑）+ `draft_evidence_card`（实时把问清楚的事实沉淀为右栏草稿）→ 结束后生成总结，用户确认后写回证据卡 |
| `evidence`（整理我的经历） | 证据卡列表，按状态筛选（全部/已确认/待确认/证据不足），卡内可编辑各字段，可标记已确认；入口链接到导入/访谈 |
| `jobs`（找目标岗位 / 求职进度，两个导航项指向同一页面） | 手动新增岗位（公司+岗位+JD）；导入 Boss 直聘爬虫产物 JSON（文件或粘贴，跨文件按 id 合并，缺 JD 详情如实标注，去重）；「准备优先级」面板：批量分析所有未分析岗位（Agent，带进度），按证据覆盖度排序并标「先准备这个」；每张岗位卡可用 ‹ › 推进求职进度状态（收藏→准备投递→已投递→已回复→面试中→Offer/拒绝） |
| `pkg`（准备这个岗位） | 核心枢纽页：JD 粘贴/编辑 + 分析；五步准备流水线（岗位分析/定制简历/面试问题/模拟面试/面试后复盘，每步可点击跳转、带完成状态与告警）；JD 拆解视图（核心职责/必要能力/加分能力/隐含要求）；证据匹配视图（覆盖度/证据强度/清晰度/风险项四指标 + 强/弱/未匹配三栏 + 建议弱化 + 禁止夸大风险项）；「一键准备这个岗位」Agent 编排（分析→简历→暂停等用户确认 AI 建议→问题），全程可见进度条；迭代记录时间线（历次模拟/复盘产出的建议） |
| `resume`（定制简历，经 `pkg` 步骤条跳入） | 每条内容（bullet）关联证据、AI 改写建议可接受/拒绝/编辑、★ 面试钩子标记、易被追问提示；多版本保存/恢复；模板系统（移植自 Telos 的 `TemplateSpec` + `SpecRenderer`，14 个预设 + 自定义骨架/页眉/字号/密度/字族/强调色）；服务端 Puppeteer 真实导出 PDF（内部标注不出现在导出文件里） |
| `qa`（面试问题，经 `pkg` 步骤条跳入） | 按分类（自我介绍/项目讲述/简历追问/技术/业务/协作/风险/反问）生成问题，每题标注来源 bullet、对应岗位要求、准备状态（未准备/准备中/已掌握/高风险）、可编辑答案、深挖追问提示；简历更新后显示「已过时」提示 |
| `mock`（模拟面试） | 基于当前简历 + 面试钩子 + 历史薄弱点的多轮对话面试官；Agent 模式下面试官可调用 `search_evidence` 核实说法；支持浏览器语音输入（Web Speech API，非本地推理，Firefox 降级为打字）；结束后生成报告（较好/被问穿/不完整/需重备/简历修改建议/下次重点）；历史场次可回看 |
| `records`（面试后复盘） | 粘贴转写文本（确定性解析：识别说话人、抽取问答对、标记含糊/中断表述）或「上传录音」（**仅为演示流水线，不做真实转录**，见第 3 节）；AI 生成结构化复盘：时间轴对话、问答与追问链、钩子命中、结构化笔记、评分/结论、亮点/问题/缺口/下一轮清单；产出的修改建议需用户逐条「采纳/忽略」才会写入 QA / 简历 / 证据卡 |
| `settings`（设置） | AI 接入配置（协议、URL、模型、Key，明文存本机 localStorage，可选「仅本次使用」，连接测试）；数据管理（导出备份 JSON、导入前预览数量对比后需二次确认覆盖、两次点击确认清空恢复演示数据、显示 IndexedDB 用量估算） |
| `home` / `auth` | 营销首页（scrollytelling 叙事）+ 演示登录（无真实账号系统，「登录」和「使用演示账号」按钮效果相同，直接进入 app） |

### 1.2 跨页面的核心机制

- **本地优先持久化**：`zustand persist` + IndexedDB 适配器（`lib/storage.ts`，隐私模式回落 localStorage），只持久化领域数据与导航位置（见 `PERSIST_KEYS`），瞬态状态（加载中、聊天草稿）不落盘。无服务端数据库。
- **AI 双模式**：未配置 Key 时全站保持 mock 模式，每个 AI 动作都有「不编造事实」的确定性兜底生成器（`lib/seed.ts` 里的 `fallback*` 系列）；配置了自己的 API URL + Key 后，走 `POST /api/ai` 纯透传路由（支持 Anthropic 原生 / OpenAI 兼容双协议），Key 不落盘不记日志，失败原样透出上游报错并回落兜底。
- **Agent 能力（三处，均有确认关卡、工具调用对用户透明）**：批量岗位分析排序、带 `search_evidence`/`draft_evidence_card` 工具的访谈官与模拟面试官、一键准备编排（分析→简历→暂停确认→问题）。仅 Anthropic 原生协议支持工具调用。
- **数据反哺闭环**：简历改动 → 标记相关 QA/模拟「已过时」；复盘产出的建议逐条确认后落到 QA（`origin: "real"`）/ 简历（待决策建议）/ 证据卡备注。

## 2. 当前用户完整操作流程

以「从零开始的新用户」视角走一遍现状（数据来自 `seed.ts`，实际是预置了 3 个岗位的演示账号，见第 3 节的说明）：

1. 打开首页 → 点「进入演示」/「导入简历开始」直接进入 app（`screen: "app"`），无需真实注册登录。
2. 落在「开始准备」（Dashboard）：看到首次引导的 8 步 chips 和「下一步」黑色行动卡（此时因为演示数据已有一条待确认的复盘建议，会直接提示「处理 3 条复盘建议」）。
3. 通过侧边栏「整理我的经历」进入证据库，点「导入旧简历」→ 粘贴/示例简历 → AI 拆解 → 转为证据卡（状态：待确认）。
4. 可选：从证据卡点「访谈补全 →」或独立发起「AI 访谈补全经历」，多轮追问后生成总结，确认写回证据卡（状态可能变为已确认）。
5. 通过「找目标岗位」进入岗位列表，新增岗位（粘贴 JD）或导入 Boss 直聘产物，创建后自动跳转「准备这个岗位」。
6. 在「准备这个岗位」：粘贴/确认 JD → 点「分析 JD」（或用「一键准备这个岗位」自动编排）→ 看到岗位拆解 + 证据匹配结果 → 沿五步流水线点「定制简历」。
7. 在简历编辑器中逐条查看证据关联、接受/拒绝/编辑 AI 建议、标记面试钩子、可选切换模板/导出 PDF。
8. 回到「准备这个岗位」或直接跳转，点「面试问题」生成问题列表，逐题标记准备状态、编辑答案。
9. 进入「模拟面试」，与面试官多轮对话（可能被追问面试钩子和历史薄弱点），结束后看报告，报告里的建议可跳去修改简历/问题。
10. 真实面试结束后，进入「面试后复盘」粘贴转写文本（或点「上传录音」体验演示流水线），得到结构化复盘，逐条确认建议是否采纳。
11. 采纳的建议自动更新到 QA/简历/证据卡；「开始准备」的下一步引擎重新计算，引导用户开始准备下一个岗位或补齐当前岗位的缺口。
12. 全程可在「设置」里换成自己的 AI Key，或导出/导入备份做设备迁移。

这条路径目前**是可以走通的**（`STATUS.md` 记录了完整走查通过），但下面第 3 节列出了沿途会增加认知负担或造成困惑的地方。

## 3. 流程中断点、重复功能和理解成本高的地方

按严重程度排列：

### 3.1 「找目标岗位」与「求职进度」是同一个页面的两个入口

导航改名后，侧边栏里「找目标岗位」→ `jobs` 和「求职进度」→ `jobs` 指向完全相同的 tab、完全相同的内容，中间没有任何视图差异（既不是不同筛选，也不是不同排序）。这与 `STATUS.md` 明确记录的产品决策矛盾——v2 阶段**刻意删除**了独立的「求职进度看板」页面（8 个页面收敛原则）。当前的双入口是本次导航改名时为了照顾「新用户找岗位」与「老用户看进度」两种心智做的权宜之计，但代价是用户点两个不同的导航项、得到完全一样的页面，会怀疑自己是不是点错了。**这是第一阶段最优先要处理的问题。**

### 3.2 「准备这个岗位」页面信息密度过高，首屏认知负担大

该页面现在一次性堆叠了：JD 输入框、五步流水线卡、（可能出现的）一键准备进度条/暂停提示、JD 拆解三栏+隐含要求、证据匹配四指标深色面板+强/弱/未匹配三栏+建议弱化+风险表述、迭代记录时间线。对第一次进入这个页面的用户来说，「现在在哪、下一步做什么」这两个问题被淹没在大量分析细节里，需要向下滚动很久。五步流水线卡本来是很好的「总览」设计，但紧接着强制展开的详细分析内容削弱了它的总览效果。

### 3.3 「求职进度」状态与「准备完成度」是两套并行的进度模型，容易混淆

`Job.status`（收藏/准备投递/已投递/已回复/面试中/Offer/拒绝，代表**投递进度**）和 `pkg` 页面五步流水线状态（岗位分析/简历/问题/模拟/复盘，代表**准备完成度**）是两个独立维度，但在 Jobs 卡片、Dashboard「当前正在准备的岗位」列表里被拼接展示成一行文字（如「面试中 · 准备中 · 当前」），没有视觉上区分「这是投递进度」还是「这是准备进度」。新用户第一次看到容易以为是同一件事。

### 3.4 证据补充有两个独立入口，选择成本高

「导入旧简历」（批量、整段文本 → 多条新证据）和「AI 访谈补全经历」（单个项目、深度追问 → 新证据或补全已有证据的空字段）功能上互补，但页面上是两个平级按钮，且从证据库以外的入口很难判断该用哪个。目前靠证据库空状态里的引导文案兜底，但没有更明确的「先做哪个」引导。

### 3.5 「上传录音」功能名不副实

`Records.tsx` 的「点击上传面试录音（mp3/m4a/wav）」在代码里完全不读取所选文件内容——`uploadRecording` 只是用 `setTimeout` 模拟四段进度条，最终套用一段写死的示例转写文本生成复盘。这在演示模式下是合理的（且已用小字标注「演示模式：接入转写服务后为真实内容」），但入口文案本身（“点击上传面试录音”）会让用户误以为已经支持真实音频转录，直到看完流程才会注意到那行小字免责声明。这是一个诚实性问题，优先级应该提前到第一阶段解决（哪怕只是调整入口文案的显著程度）。

### 3.6 Dashboard 的「首次引导」与「下一步引擎」信息重叠

首次进入时，Dashboard 同时显示 8 步引导 chips（点了就关闭的完整闭环导览）和黑色「下一步」行动卡（当前最该做的一件事）。两者经常在指向同一个动作（比如都建议去导入简历），关闭引导条后完全靠「下一步」卡片继续引导，但在关闭之前两套文案同时出现，可能让用户不确定该看哪个。

### 3.7 数据模型层面的隐性耦合（记录性问题，本轮不改）

`ResumeBullet.ev` 用证据**标题字符串**而非证据 id 来关联证据卡（`types.ts` 里 `ev: string | null`），如果用户日后编辑证据卡标题，已生成简历里的关联会静默失效（不会报错，只是匹配不上）。这是现有数据结构的选择，本文档只记录、不在本轮改动。

### 3.8 证据匹配结果（Match）没有被结构化地喂给下一步生成

`pkg` 页面生成的「证据匹配」（强/弱/未匹配、建议弱化、风险表述）目前只是给用户看的分析结果，`generateResume`/`generateQa` 的 AI 提示词并不会把这份 `match` 结构化地传入——它们各自独立地拿 JD 原文 + 证据库重新生成一遍。用户体感上会觉得「明明分析过了，为什么生成简历时感觉没有用上」。这是一个未来可以做但**不属于本轮结构性改动**的点，先记录。

## 4. 建议的用户主流程

```
整理经历 → 添加目标岗位 → 准备该岗位 → 模拟面试 → 真实面试复盘 → 更新下一轮准备
   ↑__________________________________________________________________|
```

这条主流程和当前已完成的导航改名（`整理我的经历 / 找目标岗位 / 准备这个岗位 / 模拟面试 / 面试后复盘`）在命名上已经对齐，本节把每一步的**进入条件**和**退出条件**定义清楚，作为后续调整页面内容的依据：

| 步骤 | 进入条件 | 页面在这一步该回答的问题 | 退出条件（可以进入下一步） |
|---|---|---|---|
| 整理经历 | 用户没有或想补充经历 | 「我做过什么、哪些已经想清楚了」 | 至少有一条经历状态为「已确认」 |
| 添加目标岗位 | 已有可用经历 | 「我要准备投哪个岗位」 | 至少创建 1 个岗位并粘贴了 JD |
| 准备该岗位 | 已选定岗位 | 「这个岗位我准备到哪一步了、下一步做什么」 | 简历 + 面试问题都已生成 |
| 模拟面试 | 该岗位简历已生成 | 「面试官会怎么问、我接得住吗」 | 至少完成一场模拟并看过报告 |
| 真实面试复盘 | 用户参加了真实面试 | 「这场面试暴露了什么、要改什么」 | 复盘建议全部「采纳」或「忽略」 |
| 更新下一轮准备 | 复盘产生了建议 | 「哪些证据/简历/问题需要更新」 | 回到「准备该岗位」形成闭环，或去「添加目标岗位」准备下一个 |

「更新下一轮准备」不是一个独立页面，而是一种**状态**——由复盘建议、简历改动引发的 `qaStale`/`mockStale` 标记来驱动 Dashboard 的下一步引擎重新指向对应步骤。这与当前实现的机制一致，不需要新增页面。

## 5. 现有页面与新流程的映射关系

| 主流程步骤 | 现有页面（tab） | 现状 | 备注 |
|---|---|---|---|
| 整理经历 | `evidence`（枢纽）+ `import` + `interview`（子步骤） | 已对齐 | `import`/`interview` 不在侧边栏，只能从 `evidence` 或 Dashboard 顶部按钮进入，符合「收敛为子步骤」的设计 |
| 添加目标岗位 | `jobs` | 已对齐 | 见 3.1，与「求职进度」重复入口需要合并 |
| 准备该岗位 | `pkg`（枢纽）+ `resume` + `qa`（步骤条跳入） | 已对齐但信息过密 | 见 3.2 |
| 模拟面试 | `mock` | 已对齐 | 无需调整 |
| 真实面试复盘 | `records` | 已对齐但入口文案有诚实性问题 | 见 3.5 |
| 更新下一轮准备 | 无独立页面，由 `qaStale`/`mockStale` + `records` 的 suggestions + Dashboard 的 `useNextAction` 共同承担 | 已对齐 | 不需要新页面 |
| （支撑）总览 | `dashboard` | 已对齐但与首次引导有信息重叠 | 见 3.6 |
| （支撑）配置 | `settings` | 已对齐 | 无需调整 |

**结论：现有页面结构已经基本贴合建议的主流程，不需要新增或推翻任何页面。** 本轮工作的重点是解决第 3 节列出的重复入口、信息密度和诚实性问题，而不是重新设计信息架构。

## 6. 建议保留、调整、合并和暂缓的功能

### 保留（现状正确，不动）

- 「一个岗位对应一套专属简历、QA、模拟面试和复盘」的核心数据模型（`Record<jobId, T>` 结构）
- 证据库作为全局共享资产、跨岗位复用的设计

## 7. 核心数据对象及其关联关系

```
Evidence（职业经历/证据卡，全局共享，不属于任何岗位）
  id, title, project, background, responsibilities[], actions[],
  challenges[], collaboration?, results[], skills[], roles[],
  source, status(confirmed|pending|insufficient), note?

Job（岗位申请包的锚点）
  id, company, role, status(求职进度七态), match, jd
    │
    ├─ Analysis        1:1   （JD 拆解：职责/必要/加分/隐含要求）
    ├─ Match           1:1   （证据匹配：四指标 + 强/弱/未匹配 + 弱化建议 + 风险表述；
    │                          Match.strong/weak/none[].ev 引用 Evidence.title）
    ├─ Resume          1:1（当前版本）
    │     └─ ResumeExp[] → ResumeBullet[]
    │           ResumeBullet.ev 引用 Evidence.title（字符串关联，见 3.7）
    ├─ ResumeVersion[] 1:N （简历历史快照）
    ├─ QaItem[]        1:N （面试问题，fromBullet 引用 ResumeBullet.id）
    ├─ MockSession[]   1:N （模拟面试场次，含 MockReport）
    └─ InterviewRecord[] 1:N （真实面试复盘）
          ├─ TranscriptSeg[]（时间轴对话，含 flags）
          ├─ RecordQa[]（问答对，fromResume 引用 ResumeBullet.id）
          └─ RecordSuggestion[]（target: qa|resume|evidence，
                                  确认后分别写入 QaItem[] / ResumeBullet / Evidence.note）

TemplateSpec（简历排版参数，与内容完全解耦，不挂在 Job 下，是 resumeTpl/resumeSpec 全局单值）
```

关键关系说明：

- **Evidence 是唯一的「真相来源」**：所有岗位的简历、问题都从 Evidence 派生，不会反向修改 Evidence（除非通过访谈/复盘的确认流程）。
- **Job 与其衍生对象是 1:N 的 `Record<jobId, T>` 映射**，不是嵌套对象——这保证了「一个岗位对应一套专属简历/QA/模拟/复盘」互不干扰。
- **ResumeBullet 与 Evidence、QaItem 与 ResumeBullet、RecordQa/RecordSuggestion 与 ResumeBullet** 之间都是**字符串引用**（标题或 id），不是数据库外键约束，改名/删除源对象不会级联报错，只会静默失去关联——这是当前架构下需要注意但本轮不处理的脆弱点。

## 8. 本地存储与后端能力的边界

### 本地（用户浏览器）

| 存储 | 内容 | 机制 |
|---|---|---|
| IndexedDB | 全部领域数据：evidence / jobs / analyses / matches / resumes / resumeVersions / resumeTpl / resumeSpec / qa / qaStale / mocks / mockStale / records / 导航位置 | `zustand persist` + `lib/storage.ts` 适配器，隐私模式回落 localStorage |
| localStorage | AI 接入配置（protocol/url/model/key/remember） | 独立 store（`lib/aiConfig.ts`），与主数据分开，`remember=false` 时 Key 不落盘 |
| 无 | 用户账号、多设备同步 | `auth` 页是演示登录，没有真实身份系统；迁移只能靠手动导出/导入备份 JSON |

### 后端（当前仅 2 个无状态 API route）

| 接口 | 职责 | 边界 |
|---|---|---|
| `POST /api/ai` | 纯透传代理，把用户自己的 AI Key 转发到 Anthropic 原生或 OpenAI 兼容端点；支持 Agent 工具循环（tool_use/tool_result 透传） | 不落盘、不记日志、不做平台自己出资的模型代理；没有鉴权、没有限流、没有用量统计 |
| `POST /api/export/pdf` | 服务端 Puppeteer + **本机 Chrome** 把简历 DOM 渲染成 PDF | 依赖运行 Next.js 的机器上能找到 Chrome/Chromium（系统安装或 Playwright/Puppeteer 缓存），**如果部署到没有 Chrome 的无服务器环境会直接失败**——这是当前架构里唯一对「运行环境」有强假设的服务端能力，值得在后续部署阶段重点关注 |

### 与用户设定的后端边界原则对照

用户要求「后端只承担必要的模型代理、转录、鉴权、限流和用量统计」，现状：

- ✅ 模型代理：已具备（透传形态），❌ 但不支持平台托管调用/计量的模式（用户目前必须自己有 Key）
- ❌ 转录：完全未实现（`records` 页的「上传录音」是前端模拟）
- ❌ 鉴权：完全未实现（无用户系统、无 session、无多用户隔离）
- ❌ 限流：未实现
- ❌ 用量统计：未实现（仅有单次调用失败原因展示，无累计统计）

这四项缺口是「本地优先 + 最小后端」架构下需要补的部分，但都涉及新增依赖/新增服务，不在本轮（不新增依赖）范围内，见第 9 节的分阶段安排。
