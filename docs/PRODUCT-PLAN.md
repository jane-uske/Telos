## 1. 当前产品已经实现的功能

产品定位：帮工程师把零散经历整理成**可追溯的职业证据**，再针对具体岗位生成可信简历与求职材料，覆盖「经历 → 岗位 → 简历/问题 → 模拟 → 真实面试 → 复盘反哺」的闭环。技术栈：Next.js 16 App Router + React 19 + TypeScript，单一 Zustand store（`web/src/lib/store.ts`）驱动状态，路由用 `screen`/`tab` 两级 store 状态模拟（非真实 URL 路由），IndexedDB 本地持久化。

### 1.1 已实现的页面能力（11 个 tab + 首页/登录）

| 页面（tab） | 核心能力 |
|---|---|
| `dashboard`（开始准备） | 问候语、首次引导 chips（8 步，仅首次展示）、「下一步」行动引擎（`useNextAction`，按优先级判断当前岗位最该做的一件事）、4 项统计卡、当前岗位列表、最近一次复盘摘要、待完善经历、待准备问题 |
| `import`（导入旧简历） | 粘贴/填示例简历文本 → AI（或确定性兜底）拆解成结构化经历段 → 逐条「转为证据卡」加入证据库 |
| `interview`（AI 访谈补全经历） | 选择一张未确认证据卡或发起「自由访谈」→ 多轮连续追问（背景/职责边界/关键决策/难点/协作/量化结果/证据）→ 支持 Agent 模式：`search_evidence`（核实说法有无证据支撑）+ `draft_evidence_card`（实时把问清楚的事实沉淀为右栏草稿）→ 结束后生成总结，用户确认后写回证据卡 |
| `evidence`（整理我的经历） | 证据卡列表，按状态筛选（全部/已确认/待确认/证据不足），卡内可编辑各字段，可标记已确认；入口链接到导入/访谈 |
| `jobs`（找目标岗位 / 求职进度，两个导航项指向同一页面） | 手动新增岗位（公司+岗位+JD）；导入 Boss 直聘爬虫产物 JSON（文件或粘贴，跨文件按 id 合并，缺 JD 详情如实标注，去重）；「准备优先级」面板：批量分析所有未分析岗位（Agent，带进度），按证据覆盖度排序并标「先准备这个」；每张岗位卡可用 ‹ › 推进求职进度状态（收藏→准备投递→已投递→已回复→面试中→Offer/拒绝） |
| `pkg`（准备这个岗位） | 核心枢纽页：JD 粘贴/编辑 + 分析；五步准备流水线（岗位分析/定制简历/面试问题/模拟面试/面试后复盘，每步可点击跳转、带完成状态与告警）；JD 拆解视图（核心职责/必要能力/加分能力/隐含要求）；证据匹配视图（覆盖度/证据强度/清晰度/风险项四指标 + 强/弱/未匹配三栏 + 建议弱化 + 禁止夸大风险项）；「一键准备这个岗位」Agent 编排（分析→简历→暂停等用户确认 AI 建议→问题），全程可见进度条；迭代记录时间线（历次模拟/复盘产出的建议） |
| `resume`（定制简历，经 `pkg` 步骤条跳入） | 每条内容（bullet）关联证据、AI 改写建议可接受/拒绝/编辑、★ 面试钩子标记、易被追问提示；多版本保存/恢复；模板系统（移植自 Telos 的 `TemplateSpec` + `SpecRenderer`，14 个预设 + 自定义骨架/页眉/字号/密度/字族/强调色）；服务端 Puppeteer 真实导出 PDF（内部标注不出现在导出文件里） |
| `qa`（面试问题，经 `pkg` 步骤条跳入） | 按分类（自我介绍/项目讲述/简历追问/技术/业务/协作/风险/反问）生成问题，每题标注来源 bullet、对应岗位要求、准备状态（未准备/准备中/已掌握/高风险）、可编辑答案、深挖追问提示；简历更新后显示「已过时」提示 |
| `mock`（模拟面试） | 基于当前简历 + 面试钩子 + 历史薄弱点的多轮对话面试官；Agent 模式下面试官可调用 `search_evidence` 核实说法；支持浏览器语音输入（Web Speech API，非本地推理，Firefox 降级为打字）；结束后生成报告（较好/被问穿/不完整/需重备/简历修改建议/下次重点）；历史场次可回看 |
| `records`（面试后复盘） | 粘贴转写文本（确定性解析：识别说话人、抽取问答对、标记含糊/中断表述）或「上传录音」（**仅为演示流水线，不做真实转录**，见第 3 节）；AI 生成结构化复盘：时间轴对话、问答与追问链、钩子命中、结构化笔记、评分/结论、亮点/问题/缺口/下一轮清单；产出的修改建议需用户逐条「采纳/忽略」才会写入 QA / 简历 / 证据卡 |
| `settings`（设置） | AI 接入配置（协议、URL、模型、Key，明文存本机 localStorage，可选「仅本次使用」，连接测试）；数据管理（导出备份 JSON、导入前预览数量对比后需二次确认覆盖、两次点击确认清空恢复演示数据、显示 IndexedDB 用量估算） |
| `home` / `auth` | 营销首页（scrollytelling 叙事）+ 演示登录（无真实账号系统，「登录」和「使用演示账号」按钮效果相同，直接进入 app） |

### 1.2 跨页面的核心机制

- **本地优先持久化**：`zustand persist` + IndexedDB 适配器（`lib/storage.ts`，隐私模式回落 localStorage），只持久化领域数据与导航位置（见 `PERSIST_KEYS`），瞬态状态（加载中、聊天草稿）不落盘。无服务端数据库。
- **AI 双模式**：未配置 Key 时全站保持 mock 模式，每个 AI 动作都有「不编造事实」的确定性兜底生成器（`lib/seed.ts` 里的 `fallback*` 系列）；配置了自己的 API URL + Key 后，走 `POST /api/ai` 纯透传路由（支持 Anthropic 原生 / OpenAI 兼容双协议），Key 不落盘不记日志，失败原样透出上游报错并回落兜底。
- **Agent 能力（三处，均有确认关卡、工具调用对用户透明）**：批量岗位分析排序、带 `search_evidence`/`draft_evidence_card` 工具的访谈官与模拟面试官、一键准备编排（分析→简历→暂停确认→问题）。仅 Anthropic 原生协议支持工具调用。
- **数据反哺闭环**：简历改动 → 标记相关 QA/模拟「已过时」；复盘产出的建议逐条确认后落到 QA（`origin: "real"`）/ 简历（待决策建议）/ 证据卡备注。

## 2. 当前用户完整操作流程

以「从零开始的新用户」视角走一遍现状（数据来自 `seed.ts`，实际是预置了 3 个岗位的演示账号，见第 3 节的说明）：

1. 打开首页 → 点「进入演示」/「导入简历开始」直接进入 app（`screen: "app"`），无需真实注册登录。**注意：这一步目前无论点哪个按钮都会加载 `seed.ts` 演示数据，真实新用户拿不到空白起点（见 3.1，这是本文档调整后的第一优先级问题）。**
2. 落在「开始准备」（Dashboard）：看到首次引导的 8 步 chips 和「下一步」黑色行动卡（此时因为演示数据已有一条待确认的复盘建议，会直接提示「处理 3 条复盘建议」）。
3. 通过侧边栏「整理我的经历」进入证据库，点「导入旧简历」→ 粘贴/示例简历 → AI 拆解 → 转为证据卡（状态：待确认）。
4. 可选：从证据卡点「访谈补全 →」或独立发起「AI 访谈补全经历」，多轮追问后生成总结，确认写回证据卡（状态可能变为已确认）。
5. 通过「找目标岗位」进入岗位列表，新增岗位（粘贴 JD）或导入 Boss 直聘产物，创建后自动跳转「准备这个岗位」。
6. 在「准备这个岗位」：粘贴/确认 JD → 点「分析 JD」（或用「一键准备这个岗位」自动编排）→ 看到岗位拆解 + 证据匹配结果 → 沿五步流水线点「定制简历」。
7. 在简历编辑器中逐条查看证据关联、接受/拒绝/编辑 AI 建议、标记面试钩子、可选切换模板/导出 PDF。
8. 回到「准备这个岗位」或直接跳转，点「面试问题」生成问题列表，逐题标记准备状态、编辑答案。
9. 进入「模拟面试」，与面试官多轮对话（可能被追问面试钩子和历史薄弱点），结束后看报告，报告里的建议可跳去修改简历/问题。
10. 真实面试结束后，进入「面试后复盘」粘贴转写文本（或点「上传录音」体验演示流水线），得到结构化复盘，逐条确认建议是否采纳。
11. 采纳的建议自动更新到 QA/简历/证据卡；「开始准备」的下一步引擎重新计算，引导用户开始准备下一个岗位或补齐当前岗位的缺口。
12. 全程可在「设置」里换成自己的 AI Key，或导出/导入备份做设备迁移。

这条路径目前**是可以走通的**（`STATUS.md` 记录了完整走查通过），但下面第 3 节列出了沿途会增加认知负担或造成困惑的地方，按调整后的优先级排列。

## 3. 流程中断点、重复功能和理解成本高的地方

按调整后的严重程度排列：

### 3.1 演示模式与真实新用户状态未分离（最高优先级）

当前「登录」和「使用演示账号」两个按钮效果完全相同——不管点哪个，都会直接进入已经预置了 3 个岗位、多条证据、待处理复盘建议的 `seed.ts` 演示数据（见 1.1 `home`/`auth` 行、第 2 节步骤 1）。这带来两个问题：

1. **真实新用户永远看不到「从 0 开始」的产品**：所有首次引导（8 步 chips）、下一步引擎的判断，都是在已经有数据的状态下计算出来的，无法验证「真正空白」时的引导是否合理。
2. **演示数据和用户自己的数据混在一起**：用户如果直接开始添加自己的真实岗位/经历，会和演示岗位混杂在同一个列表里，不清楚哪些是示例、哪些是自己的。

**这是本文档调整后的第一优先级问题**：「开始使用」必须让用户进入真正的空白状态（0 证据、0 岗位）；「查看演示」应该是一个独立、显式的可选动作，用户主动点了才加载种子数据，不点则保持空白。这个问题不解决，后面所有关于「首次引导」「下一步引擎」的判断都建立在失真的前提上。

### 3.2 「找目标岗位」与「求职进度」不应被简单合并删除概念，而应做成单入口双视图

导航改名后，侧边栏里「找目标岗位」→ `jobs` 和「求职进度」→ `jobs` 指向完全相同的 tab、完全相同的内容，中间没有任何视图差异（既不是不同筛选，也不是不同排序）。这与 `STATUS.md` 明确记录的产品决策矛盾——v2 阶段**刻意删除**了独立的「求职进度看板」页面（8 个页面收敛原则）。

但简单删除其中一个导航项、只留「找目标岗位」，会丢掉「求职进度」这个用户心智真实存在的概念——老用户想看「我投递到哪一步了」时，找不到对应入口。**调整后的方案**：把两个导航项合并为一个「岗位与进度」入口，页面内部通过两个视图（tab/分段控件）区分：「找目标岗位」（浏览、新增、分析岗位，即现有 `jobs` 页大部分内容）和「我的申请」（按 `Job.status` 投递进度分组展示，承接原本「求职进度」的心智，而不是把这个概念直接删掉）。

### 3.3 「准备这个岗位」页面信息密度过高，首屏认知负担大

该页面现在一次性堆叠了：JD 输入框、五步流水线卡、（可能出现的）一键准备进度条/暂停提示、JD 拆解三栏+隐含要求、证据匹配四指标深色面板+强/弱/未匹配三栏+建议弱化+风险表述、迭代记录时间线。对第一次进入这个页面的用户来说，「现在在哪、下一步做什么」这两个问题被淹没在大量分析细节里，需要向下滚动很久。五步流水线卡本来是很好的「总览」设计，但紧接着强制展开的详细分析内容削弱了它的总览效果。

### 3.4 「求职进度」状态与「准备完成度」是两套并行的进度模型，容易混淆

`Job.status`（收藏/准备投递/已投递/已回复/面试中/Offer/拒绝，代表**投递进度**）和 `pkg` 页面五步流水线状态（岗位分析/简历/问题/模拟/复盘，代表**准备完成度**）是两个独立维度，但在 Jobs 卡片、Dashboard「当前正在准备的岗位」列表里被拼接展示成一行文字（如「面试中 · 准备中 · 当前」），没有视觉上区分「这是投递进度」还是「这是准备进度」。3.2 合并为「我的申请」视图后，这个问题依然存在，需要单独处理（见阶段 2）。

### 3.5 「整理我的经历」需要明确区分两个入口：有简历 vs 没简历

「导入旧简历」（批量、整段文本 → 多条新证据）和「AI 访谈补全经历」（单个项目、深度追问 → 新证据或补全已有证据的空字段）功能上互补，但目前只是页面上两个平级按钮，文案没有讲清楚「你有没有现成简历」这个真正的分流条件，从证据库以外的入口（比如 Dashboard 首次引导）也很难判断该用哪个。**调整后的方案**：把两个入口明确重新表述为「上传已有简历」（对应现有 `import`）与「没有简历？用 AI 从零帮你整理」（对应现有 `interview` 自由访谈模式），让用户凭「我有没有简历」这一个问题就能决定走哪条路。

### 3.6 「整理我的经历」目前不支持任何文件上传，PDF/Word 均不支持

`Import.tsx` 当前只有「粘贴/填示例简历文本」的文本框，没有真正的文件上传控件。这在 3.5 调整入口文案为「上传已有简历」之后会形成新的诚实性问题（与 3.7「上传录音」同类）——用户看到「上传」字样，很可能会尝试拖入 PDF 文件，而当前完全没有解析能力。至少需要在用户操作之前就说明当前只支持粘贴纯文本、暂不支持 PDF/Word；如果本阶段愿意接受一次新增依赖的例外，也可以直接引入浏览器端 PDF 文本提取实现真正的上传解析。

### 3.7 「上传录音」功能名不副实

`Records.tsx` 的「点击上传面试录音（mp3/m4a/wav）」在代码里完全不读取所选文件内容——`uploadRecording` 只是用 `setTimeout` 模拟四段进度条，最终套用一段写死的示例转写文本生成复盘。这在演示模式下是合理的（且已用小字标注「演示模式：接入转写服务后为真实内容」），但入口文案本身（"点击上传面试录音"）会让用户误以为已经支持真实音频转录，直到看完流程才会注意到那行小字免责声明。这是一个诚实性问题，需要提前到用户点击前就能看到。

### 3.8 Dashboard 的「首次引导」与「下一步引擎」信息重叠

首次进入时，Dashboard 同时显示 8 步引导 chips（点了就关闭的完整闭环导览）和黑色「下一步」行动卡（当前最该做的一件事）。两者经常在指向同一个动作（比如都建议去导入简历），关闭引导条后完全靠「下一步」卡片继续引导，但在关闭之前两套文案同时出现，可能让用户不确定该看哪个。**注意**：这个问题的准确判断依赖 3.1 先落地——只有演示数据和真实数据分开后，「首次引导」的触发条件才能准确对应「真的什么都没有」，而不是「演示数据里凑巧有个待办」。

### 3.9 Evidence / ResumeBullet / Match 等对象间使用字符串而非稳定 ID 关联

`ResumeBullet.ev` 用证据**标题字符串**而非证据 id 来关联证据卡（`types.ts` 里 `ev: string | null`），`Match.strong/weak/none[].ev` 同样引用 `Evidence.title`。如果用户日后编辑证据卡标题，已生成简历/问题里的关联会静默失效（不会报错，只是匹配不上）。这是现有数据结构的选择，**已排入阶段 2**（原先只作记录、不在本轮改动，现在正式列入实施计划，见第 9、11 节）。

### 3.10 证据匹配结果（Match）没有被结构化地喂给下一步生成

`pkg` 页面生成的「证据匹配」（强/弱/未匹配、建议弱化、风险表述）目前只是给用户看的分析结果，`generateResume`/`generateQa` 的 AI 提示词并不会把这份 `match` 结构化地传入——它们各自独立地拿 JD 原文 + 证据库重新生成一遍。用户体感上会觉得「明明分析过了，为什么生成简历时感觉没有用上」。**已排入阶段 2**（原先记录为"未来优化、非结构性调整"，现在正式列入实施计划，见第 9、11 节）。

## 4. 建议的用户主流程

```
（首次进入：空白状态，查看演示为可选动作，见 3.1）
   ↓
整理经历 → 添加目标岗位 → 准备该岗位 → 模拟面试 → 真实面试复盘 → 更新下一轮准备
   ↑__________________________________________________________________|
```

这条主流程和当前已完成的导航改名（`整理我的经历 / 找目标岗位 / 准备这个岗位 / 模拟面试 / 面试后复盘`）在命名上已经对齐，本节把每一步的**进入条件**和**退出条件**定义清楚，作为后续调整页面内容的依据：

| 步骤 | 进入条件 | 页面在这一步该回答的问题 | 退出条件（可以进入下一步） |
|---|---|---|---|
| 整理经历 | 用户没有或想补充经历（真实新用户应从 0 条开始，而非预置的演示数据，见 3.1） | 「我做过什么、哪些已经想清楚了」 | 至少有一条经历状态为「已确认」 |
| 添加目标岗位 | 已有可用经历 | 「我要准备投哪个岗位」 | 至少创建 1 个岗位并粘贴了 JD |
| 准备该岗位 | 已选定岗位 | 「这个岗位我准备到哪一步了、下一步做什么」 | 简历 + 面试问题都已生成 |
| 模拟面试 | 该岗位简历已生成 | 「面试官会怎么问、我接得住吗」 | 至少完成一场模拟并看过报告 |
| 真实面试复盘 | 用户参加了真实面试 | 「这场面试暴露了什么、要改什么」 | 复盘建议全部「采纳」或「忽略」 |
| 更新下一轮准备 | 复盘产生了建议 | 「哪些证据/简历/问题需要更新」 | 回到「准备该岗位」形成闭环，或去「添加目标岗位」准备下一个 |

「更新下一轮准备」不是一个独立页面，而是一种**状态**——由复盘建议、简历改动引发的 `qaStale`/`mockStale` 标记来驱动 Dashboard 的下一步引擎重新指向对应步骤。这与当前实现的机制一致，不需要新增页面。

## 7. 核心数据对象及其关联关系

```
Evidence（职业经历/证据卡，全局共享，不属于任何岗位）
  id, title, project, background, responsibilities[], actions[],
  challenges[], collaboration?, results[], skills[], roles[],
  source, status(confirmed|pending|insufficient), note?

Job（岗位申请包的锚点）
  id, company, role, status(求职进度七态), match, jd
    │
    ├─ Analysis        1:1   （JD 拆解：职责/必要/加分/隐含要求）
    ├─ Match           1:1   （证据匹配：四指标 + 强/弱/未匹配 + 弱化建议 + 风险表述；
    │                          Match.strong/weak/none[].ev 引用 Evidence.title，
    │                          计划阶段 2 改为引用 Evidence.id，见 3.9）
    ├─ Resume          1:1（当前版本）
    │     └─ ResumeExp[] → ResumeBullet[]
    │           ResumeBullet.ev 引用 Evidence.title（字符串关联，
    │           计划阶段 2 改为引用 Evidence.id，见 3.9）
    ├─ ResumeVersion[] 1:N （简历历史快照）
    ├─ QaItem[]        1:N （面试问题，fromBullet 引用 ResumeBullet.id）
    ├─ MockSession[]   1:N （模拟面试场次，含 MockReport）
    └─ InterviewRecord[] 1:N （真实面试复盘）
          ├─ TranscriptSeg[]（时间轴对话，含 flags）
          ├─ RecordQa[]（问答对，fromResume 引用 ResumeBullet.id）
          └─ RecordSuggestion[]（target: qa|resume|evidence，
                                  确认后分别写入 QaItem[] / ResumeBullet / Evidence.note）

TemplateSpec（简历排版参数，与内容完全解耦，不挂在 Job 下，是 resumeTpl/resumeSpec 全局单值）
```

关键关系说明：

- **Evidence 是唯一的「真相来源」**：所有岗位的简历、问题都从 Evidence 派生，不会反向修改 Evidence（除非通过访谈/复盘的确认流程）。
- **Job 与其衍生对象是 1:N 的 `Record<jobId, T>` 映射**，不是嵌套对象——这保证了「一个岗位对应一套专属简历/QA/模拟/复盘」互不干扰。
- **ResumeBullet 与 Evidence、Match 与 Evidence** 之间目前是**字符串引用**（标题），不是数据库外键约束，改名/删除源对象不会级联报错，只会静默失去关联——**这个脆弱点已排入阶段 2 处理**（见 3.9、第 9、11 节），需要配套一次性数据迁移。`QaItem`/`RecordQa`/`RecordSuggestion` 与 `ResumeBullet.id` 之间本来就是 id 引用，不受影响。

## 8. 本地存储与后端能力的边界

### 本地（用户浏览器）

| 存储 | 内容 | 机制 |
|---|---|---|
| IndexedDB | 全部领域数据：evidence / jobs / analyses / matches / resumes / resumeVersions / resumeTpl / resumeSpec / qa / qaStale / mocks / mockStale / records / 导航位置 | `zustand persist` + `lib/storage.ts` 适配器，隐私模式回落 localStorage |
| localStorage | AI 接入配置（protocol/url/model/key/remember） | 独立 store（`lib/aiConfig.ts`），与主数据分开，`remember=false` 时 Key 不落盘 |
| 无 | 用户账号、多设备同步 | `auth` 页是演示登录，没有真实身份系统；迁移只能靠手动导出/导入备份 JSON |

### 后端（当前仅 2 个无状态 API route）

| 接口 | 职责 | 边界 |
|---|---|---|
| `POST /api/ai` | 纯透传代理，把用户自己的 AI Key 转发到 Anthropic 原生或 OpenAI 兼容端点；支持 Agent 工具循环（tool_use/tool_result 透传） | 不落盘、不记日志、不做平台自己出资的模型代理；没有鉴权、没有限流、没有用量统计 |
| `POST /api/export/pdf` | 服务端 Puppeteer + **本机 Chrome** 把简历 DOM 渲染成 PDF | 依赖运行 Next.js 的机器上能找到 Chrome/Chromium（系统安装或 Playwright/Puppeteer 缓存），**如果部署到没有 Chrome 的无服务器环境会直接失败**——这是当前架构里唯一对「运行环境」有强假设的服务端能力，值得在后续部署阶段重点关注 |

### 与用户设定的后端边界原则对照

用户要求「后端只承担必要的模型代理、转录、鉴权、限流和用量统计」，现状与规划：

- 模型代理：✅ 已具备（BYOK 透传形态）；❌ 不支持平台托管调用/计量的模式。**规划：BYOK 保留为高级模式，平台托管 AI 作为普通用户的默认体验，排入阶段 4**（见 3.10 之后、第 9 节）
- 转录：❌ 完全未实现（`records` 页的「上传录音」是前端模拟）。**规划：排入阶段 3，优先于账号系统，不依赖鉴权即可独立上线**（见第 9 节）
- 鉴权：❌ 完全未实现（无用户系统、无 session、无多用户隔离）。**规划：排入阶段 4，仅作为多设备同步的可选锚点，不阻塞阶段 3 的转录能力**
- 限流：❌ 未实现。**规划：排入阶段 5**
- 用量统计：❌ 未实现（仅有单次调用失败原因展示，无累计统计）。**规划：排入阶段 5，为阶段 4 的平台托管 AI 计费做准备**

这四项缺口都涉及新增依赖/新增服务，不在阶段 1、2（不新增依赖）范围内，具体顺序见第 9 节。

## 9. 分阶段实施计划

原则：不推翻现有设计、不做无必要的大规模重构、不新增冗余模块；每个阶段结束都要能独立验收，且不阻塞下一阶段。**转录能力不依赖账号系统，账号与多端同步不能阻塞核心面试复盘闭环——这一点体现在阶段 3 与阶段 4 的顺序安排上。**

### 阶段 0（已完成）
导航从产品模块导向改为用户行为导向命名（PR #2）。仅记录，不重复执行。

### 阶段 1：新用户空白起点 + 清理重复入口与首屏认知负担（纯前端，除 PDF 解析例外零新增依赖）
- **演示模式与真实新用户状态分离，最高优先级**：默认空白状态，「查看演示/加载示例数据」是显式可选动作（见 3.1）
- 「找目标岗位」/「求职进度」合并为「岗位与进度」单一导航入口，页面内部保留两个视图（找目标岗位 / 我的申请），不删除任何一个概念（见 3.2）
- 「整理我的经历」明确拆分两个入口：上传已有简历 / 没有简历时用 AI 从零整理（见 3.5）
- 「整理我的经历」的 PDF/Word 上传缺口：至少在入口处显著标注当前不支持；若本阶段决定引入 `pdfjs-dist` 做浏览器端解析，则实现真正的 PDF 上传（本阶段唯一允许的依赖例外，见 3.6）
- 「准备这个岗位」页面把 JD 拆解与证据匹配收进可展开区块，首屏只保留步骤条 + 下一步（见 3.3）
- 「上传录音」入口文案与「仅演示，暂不支持真实转录」的说明提前到点击前可见（见 3.7）
- 详见第 10 节的具体文件与验收标准

### 阶段 2：数据模型稳定化 + 分析结果复用 + 进度模型视觉区分（纯前端，不引入新增运行时依赖）
- Evidence / ResumeBullet / Match 等对象间的字符串关联改为稳定 ID，含一次性数据迁移（见 3.9）
- 岗位 Match 结果正式作为简历生成、QA 生成、模拟面试的结构化输入，替代各自独立重新分析（见 3.10）
- 岗位卡片与「我的申请」视图上把「投递进度」与「准备完成度」两套状态在视觉上分开标注（见 3.4）
- Dashboard 首次引导 chips 触发条件收窄为「确无任何非种子数据」，依赖阶段 1 已完成的演示/真实数据分离（见 3.8）
- 详见第 11 节的具体文件与验收标准

### 阶段 3：真实转录能力接入（涉及新增依赖，本轮不执行，仅规划顺序；不依赖账号系统）
- 引入语音转文字服务，替换 `records` 页当前的演示流水线，让「上传录音」名副其实
- **刻意排在「账号与多用户后端基础」之前**：真实转录直接服务于「模拟面试 → 真实面试复盘」这条核心闭环，不应该等账号系统一起做才推进；实现上也不需要用户身份，可独立上线，不被阶段 4 阻塞

### 阶段 4：账号与多用户后端基础 + 平台托管 AI（涉及新增依赖，本轮不执行，仅规划顺序）
- 引入最小鉴权（例如邮箱验证码或匿名设备 ID），让本地数据可选择性地和账号绑定，账号仅作为多设备同步的可选锚点，不强制、不改变本地优先的默认体验
- 为「模型代理」新增平台托管调用的选项：**BYOK 保留为高级模式**，供有自己 Key 的用户使用，但不再是产品默认体验；**普通用户未来应无需配置任何 Key，直接使用平台托管的 AI**
- 平台托管 AI 涉及成本，通常需要账号体系承接用量归属，因此和账号基础放在同一阶段规划，但两者可分别排期，不互相阻塞

## 11. 第二阶段需要修改的具体文件和验收标准

### 涉及文件

1. `web/src/lib/types.ts` —— 把 `ResumeBullet.ev`、`Match.strong/weak/none[].ev` 等当前用字符串（证据标题）表达的关联字段，改为存 `Evidence.id`（稳定 ID）；`Job.status` 与 pkg 五步流水线状态不受影响，只动「证据关联」这一类字段。
2. `web/src/lib/store.ts` —— 补一个一次性的数据迁移函数，在 `persist` 的 `migrate`/`onRehydrateStorage` 里把旧版本本地数据里残留的字符串关联，按标题匹配现有 Evidence 尽力回填成 id（匹配不上的关联标记为失效并提示用户，而不是静默丢弃）。
3. `web/src/components/screens/Resume.tsx` / `Qa.tsx` / `Package.tsx`（`MatchView`）—— 涉及读取 `.ev` 字段做展示或再编辑的地方，改成按 id 查找 Evidence，而不是按标题字符串查找。
4. `web/src/lib/ai.ts`（或 `generateResume`/`generateQa` 所在位置）—— 把 `pkg` 页面已生成的 `Match`（强/弱/未匹配、建议弱化、风险表述）结构化地作为 prompt 输入传给简历生成、QA 生成，以及 `mock` 模拟面试官的上下文，取代目前「各自独立重新分析一遍 JD + 证据库」的做法。
5. `web/src/components/AppShell.tsx` / `screens/Jobs.tsx`（「我的申请」视图）—— 给 `Job.status`（投递进度）和 pkg 五步流水线状态（准备完成度）分别设计独立的视觉标签（不同图形/颜色/位置），不再拼接成一行文字。
6. `web/src/components/screens/Dashboard.tsx` —— 首次引导 chips 的触发条件收窄为「用户没有任何非种子数据」（依赖阶段 1 已经把种子数据和真实数据分开，这里才能准确判断）。

### 验收标准

1. 全局搜索确认 `ResumeBullet`/`Match` 结构里的证据关联字段不再是字符串标题，均为 Evidence 的 id 引用；编辑证据卡标题后，已生成简历/问题里的关联依然有效（不会静默失配）。
2. 已有本地数据的用户升级后自动完成一次性迁移，旧数据里能匹配上的关联全部转为 id，匹配不上的有明确提示而不是消失。
3. 生成简历/QA/模拟面试时，如果该岗位已经做过「证据匹配」分析，生成结果能体现出用到了强/弱匹配和风险提示的结构化信息（例如弱匹配项不会被写成过度自信的表述），而不是单独重新分析一遍 JD。
4. 岗位卡片和「我的申请」视图里，投递进度和准备完成度在视觉上一眼可辨是两件事，各自有一句说明文字。
5. `npm run build`、`npx tsc --noEmit`、`npx eslint src` 全部通过，且本阶段不引入新增运行时依赖（迁移逻辑用现有 `zustand persist` 能力实现）。
6. 人工用演示数据走一遍「编辑证据卡标题 → 检查已生成简历/QA 关联未失效」「生成简历前后对比 Match 结果是否被引用」两个场景，确认符合预期。
